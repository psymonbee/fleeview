Slide 1 · Where we are

FleetView: a live map of your AI workforce

When you give Claude Code a big job, it doesn't work alone — it hires a small team of helper AIs and hands each one a piece of the work. Normally that team is invisible. FleetView is the wall-mounted screen that shows them: who's working, on what, with which files, and whether anyone is stuck.

It runs entirely on your Mac at localhost:4747. Nothing is sent anywhere; no accounts, no cloud, no installed packages — just three small pieces reading and writing one text file.

Slide 2 · The big picture

Five stations, one relay race

Everything FleetView does is one straight line: something happens → it gets written down → the write-up is read → the picture updates on screen. Here are the five stations, in order:

Station 1 · The tap on the shoulder

Claude Code's built-in "hooks"

Claude Code has a feature where it taps a script of your choosing on the shoulder every time something happens — a session starts, a helper agent is hired, a file is edited. One block in your personal settings file points all of those taps at FleetView's note-taker. That settings block is the entire installation.

every event, as it happens

Station 2 · The note-taker & translator

A tiny script: hooks/emit-event.mjs + adapters/claude-code.mjs

The note-taker listens to each tap and passes it to a translator, which rewrites Claude Code's internal chatter into FleetView's own simple, universal language — short standard sentences like "agent demo-b1 started" or "step 3 is done". It's built to be a silent observer: if anything confuses it, it writes nothing and steps aside, so it can never interfere with your real work. The translator layer is new in v2 — it means other AI tools (like OpenAI's Codex) can be plugged in later just by writing them their own translator.

one line per event, appended

Station 3 · The diary

One plain text file: ~/.lumenade/events.jsonl

Every translated event becomes one line in this file — a running diary of everything your agents have done. It's the only thing connecting the two halves of the system, and you can open it in any text editor. Delete it and FleetView simply starts a fresh diary.

read continuously, like a ticker tape

Station 4 · The newsroom

A small server: server/server.mjs

The server watches the diary for new lines and keeps the "current picture" in its head: which sessions exist, which agents are running, what each one touched last, how the plan is progressing. Whenever the picture changes, it broadcasts the change to every open browser tab instantly — no refreshing. Start it with npm start.

live broadcast to the browser

Station 5 · The screen

The page you look at: public/index.html

One self-contained web page at localhost:4747. It draws the Orchestrator hub on the left, one card per agent on the right, animated wires between them, and the plan checklist down the side. It never talks to the internet — everything it shows comes from the newsroom next door.

Slide 3 · What you actually see

Reading the screen

Everything on the canvas answers one question at a glance: is my fleet healthy?

The hub

The orchestrator — the "manager" Claude that hires everyone else. It glows softly while a conversation turn is in progress. Since v5 it's a full card in its own right: it names the model running the show, narrates what the manager is doing, wears its own skill/tool badges, and opens its own drawer when clicked.

Agent cards

One card per helper: its name, which AI model it runs on (Claude or OpenAI, colour-coded), what it was hired to do, and a live one-line status with a ticking clock.

Plan rail (new)

The checklist down the left side. When the orchestrator writes itself a task list, each step appears here — grey waiting, pulsing in progress, ✓ done. Cards get a little S1/S2 tag showing which step they're working.

File chips (new)

Small tags on each card naming the files that agent has touched. If two running agents grab the same file, both chips pulse amber and a warning appears — your early smoke alarm for agents treading on each other.

The drawer (new)

Click any card and a panel slides out with that agent's full activity log, line by line, updating live — failures shown in red.

Final words (new)

When an agent finishes, its card keeps its parting message — the summary it handed back — so you can see what it claims to have accomplished.

Stall & error alerts (new)

An agent that's been silent for two minutes turns amber and reads "stalled". A failure makes its card flash red for a second. We discovered during testing that Claude Code stays completely silent when a tool call fails, so FleetView infers failures itself: any action that starts but never reports back is marked as failed.

Bell & narrator (new)

A bell toggle sends a desktop notification when a turn finishes while you're in another tab. And an optional narrator (off by default) has a tiny, cheap Claude write plain-English one-liners about what each agent is doing.

Slide 4 · One moment, end to end

Follow a single event through the wire

Say the orchestrator hires a Builder agent to write some code. Here's that one moment travelling the whole line — the entire trip takes about a second:

  • The event

    Claude Code spins up the Builder and taps FleetView's note-taker on the shoulder: "a new agent just started."

  • Translation

    The translator turns that into one standard line — agent id, type "builder", what it was hired to do — and appends it to the diary. (v2 got smarter here: Claude Code tells us the agent's exact id and model at hiring time, so the card is labelled precisely, never guessed.)

  • The newsroom notices

    The server, watching the diary, reads the new line, adds the Builder to its current picture, and matches its job description against the plan checklist — "Build SSE fleet server" looks like step 1, so it gets the S1 tag.

  • Broadcast

    The updated Builder record is pushed to your open browser tab over a live connection.

  • On screen

    A new card fades in, a dashed orange wire draws itself from the hub, the clock starts ticking, and the plan rail's step 1 begins to pulse. From here every file it touches and every command it runs streams onto the card as it happens.

Slide 5 · The map of the repo

What each file is for

Nine things in the repo do all the work. In plain terms:

FilePlain-English job
hooks/emit-event.mjsThe note-taker Claude Code taps on every event. Deliberately tiny and silent — it can never break your real work.
adapters/claude-code.mjsThe translator: Claude Code's dialect in, FleetView's universal language out. One of these per AI tool.
adapters/codex.mjsThe second translator (new in v3): OpenAI Codex's dialect in, same universal language out. Every Codex name gets a "codex:" prefix so the two tools can never be confused on screen.
server/enrich.mjsThe accountant (new in v3): quietly reads each agent's own session transcript on your Mac, adds up the tokens it has spent, and prices them. Nothing ever leaves your machine.
bin/install.mjsThe mover-in (new in v3): one command sets FleetView up on any Mac — copies the app to a stable home, wires the hooks in (with a backup first), and can undo itself with --uninstall.
~/.lumenade/events.jsonlThe diary (lives in your home folder, not the repo). One line per event. The whole system's single source of truth.
server/server.mjsThe newsroom: reads the diary, keeps the current picture, broadcasts changes, and serves the web page.
server/narrator.mjsThe optional commentator. Off unless you flip it on in the config.
public/index.htmlThe screen — one self-contained page, no internet access needed or used.
fleet.config.jsonThe settings dial: agent names and models, the hub's title, how long "silent" means stalled, narrator on/off — and, new in v3, the price list the accountant uses (editable when rates change).
scripts/demo.mjsA rehearsal tape: replays a fake 60-second orchestration so you can see every feature without running real agents.
docs/FLEETVIEW-SPEC.mdThe written contract. Every feature was specified here before being built; the builders and the checker all worked from this document.
test/106 automated checks (up from 33 in v2), run against real recorded Claude Code events — plus, for Codex, payload blueprints extracted from the Codex program itself, clearly labelled as such.
One clever safety rule worth knowing: the narrator uses a small Claude to describe your agents — but that Claude would itself trigger the hooks and appear on its own screen, forever. So every narrator run carries a "don't film the film crew" flag (FLEET_IGNORE), and the note-taker steps aside when it sees it. Both sides were tested with the real Claude to prove the loop can't happen.

Slide 6 · Driving it

How to use it, and how to remove it

To watch your fleet: start the server and open the page. Any Claude Code session on your Mac appears automatically the moment it does anything.

cd ~/lumenADE && npm start → then open http://localhost:4747

To see a full show without real agents: run the rehearsal tape in a second terminal — plan steps, a failure, a stall, a file collision, the lot, in about a minute.

npm run demo (add -- --fast to skip the waits)

To switch it all off: two moves, fully reversible — remove the hooks block from ~/.claude/settings.json (the taps stop), and delete the diary folder:

rm -rf ~/.lumenade

The config dial lives in fleet.config.json — rename an agent, change what counts as "stalled", tweak the token price list, or set narration.enabled to true to hear the commentator.

To set it up on another Mac (new in v3, once the repo is public): one command does the whole move-in, and the same command with --uninstall undoes it.

npx github:psymonbee/fleeview

No agents of your own yet? (new in v4) — add --with-agents and the installer also drops a starter crew into ~/.claude/agents: a builder, a checker, and a Codex relay. It never overwrites files you already have, and uninstalling leaves them with you.

Slide 7 · Proof & what's next

v4 shipped — built for people who aren't you

The shift: v3 was your window onto your fleet; v4 makes it work for strangers. If someone already has their own agents, FleetView now reads their agent files and dresses every card correctly — right name, right model, right colour — with zero configuration. If they have nothing, --with-agents hands them a starter crew. Either way, the watching part never needed their setup to match yours: that was the point all along.

The canvas grew up too: when an agent hires its own helper, the helper's card now tucks in underneath with a line drawn from its parent — a family tree, not spokes on a wheel. Before any agents exist, the hub narrates what the orchestrator itself is doing (the "getting oriented" phase you asked for), and cards sprout little badges showing which skills and outside tools each agent is using.

One honest cut: your top ask — usage limits on screen — turned out to have no data behind it: Claude Code simply doesn't write your remaining-allowance anywhere FleetView can read. Rather than show a made-up number, it's parked for v5 with the evidence written down, so nobody wastes a day rediscovering that.

The proof: 150 automated tests green, and the flagship check was live: a real agent spawned a real child during the build, and the family-tree edge drew itself on this very canvas while the child was still working. The nested-spawn behaviour wasn't guessed — it was probed with real hook captures first, same as the Codex work in v3.

Slide 8 · v5

The manager gets a card of its own

The itch: the screen looked dead until the first helper was hired — and worse, the moment the manager delegated anything, its own one-line readout was wiped, on the theory that "the fleet takes over the story". Real sessions proved that theory wrong: the manager keeps working the whole time the fleet runs. Most of a session's actual work — reading, searching, running commands, using skills and outside tools — is the manager's work, and FleetView was hiding it.

Now: the hub is the session's own card, same size as everyone else's. It's alive from the moment you hit send. When the manager hires someone, the readout now says delegate: build the SSE server instead of going blank — the moment of delegation became a visible beat in the story rather than a blackout. The card wears the same skill/tool badges the agents get, and clicking it slides out the same drawer, showing the manager's own activity log, live.

And it names names: FleetView now reads which model is actually running the show — straight off the session's own transcript, on your Mac, updating within seconds — and prints it on the hub. Not a setting, not a guess: during this very build, the canvas named the model doing the building the moment its first accounting tick ran.

The build was its own demo: v5 was built fleet-style — four builders in parallel, watched live on FleetView itself. One genuinely satisfying moment: the checking agent went silent mid-run, and its card turned amber and said "stalled" — the product flagged its own build tooling misbehaving, which is precisely the job it was built to do. 160 automated checks green (up from 153).

Honest edges: Codex-run sessions don't name their model yet (written into the contract as out of scope, not half-wired), and screenshots, video and the README's hero image are deliberately waiting on a branding-and-polish pass.
